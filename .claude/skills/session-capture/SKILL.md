# Session Capture Skill

## 你在做什么

把会话中产生的有价值信息写入 `knowledge/projects/` 下的对应项目文件夹。这是知识库的供给侧——每次有意义的会话都应该让知识库变得更完整、更有价值。

项目路径：`knowledge/projects/<project-name>/`（相对于项目根目录）

---

## 两种模式

### 模式 A：会话开始检查（CHECK）

在开始实质性工作前，用 30 秒做一个判断：

1. **读取现有项目列表**：`ls knowledge/projects/`
2. **判断当前话题**：
   - 是否匹配某个已有项目？→ 告知用户本次工作会记录到该项目
   - 是否是值得建立新项目的话题？→ 建立新项目文件夹
   - 是否是一次性问题或不值得跟踪？→ 静默跳过，不打扰用户

**何时建新项目**（满足任一条即可）：
- 话题有明确的目标和可预期的多轮工作（如：实现某功能、调研某技术、分析某投资机会）
- 涉及架构决策或技术选型，未来可能需要回溯
- 用户明确表示这是某个"项目"或"工程"

**何时不建项目**：
- 单次问答（"这段代码什么意思？"）
- 日常维护操作（"重启一下服务"）
- 已存在的成熟项目的日常迭代（直接追加到现有项目即可）

如果需要建新项目，先告诉用户："我会在 `knowledge/projects/<name>/` 下建立项目跟踪，开始工作后会自动记录。" 然后创建项目文件夹（此时不需要写文件，先工作）。

---

### 模式 B：会话内容沉淀（CAPTURE）

在会话达到沉淀条件时，提取有价值的内容写入项目文件。

#### 第一步：判断归属并确保文件夹存在

当前会话属于哪个项目？如果不确定，从已有项目列表中选最接近的，或新建一个。

**如果目标项目文件夹不存在，先创建它**（不需要写 README，那是 CHECK 模式的职责；这里只需要文件夹存在即可）。这样 CAPTURE 可以独立运作，不依赖之前是否做过 CHECK。

#### 第二步：提取要沉淀的内容

从会话中提取（跳过没有复用价值的内容）：

| 类型 | 示例 |
|------|------|
| **决策与理由** | "选择 X 方案，因为 Y；放弃 Z 因为 W" |
| **踩过的坑** | "注意：A 接口在 B 条件下会返回 C 而不是预期的 D" |
| **有效的解法** | 解决了什么问题、用的什么方法、为什么有效 |
| **发现的规律** | 系统行为的规律、代码结构的特点 |
| **未解决的问题** | 遗留的疑问、需要后续跟进的点 |
| **外部参考** | 有用的文档链接、工具、API |

**不要写**：对话的逐字记录、已经显而易见的事情、可以直接查代码的内容。

#### 第三步：写入文件

文件命名：`YYYYMMDD-<简短主题>.md`，写到对应项目文件夹。

**文件结构模板**：

```markdown
# <主题>

> 日期：YYYY-MM-DD | 会话：<一句话描述这次会话做了什么>

## 背景
<为什么做这件事，一两句话>

## 关键决策
- **决策**：<做了什么选择>
  **理由**：<为什么这么选>

## 踩坑记录
- **问题**：<描述现象>
  **原因**：<根本原因>
  **解法**：<怎么解决的>

## 有效模式
<可复用的方法论或实现模式>

## 遗留问题
- [ ] <需要后续跟进的点>

## 参考
- <有用的链接或资料>
```

根据实际内容灵活增减章节。没有内容的章节直接删掉，不要留空壳。

## 行为准则

**轻量优先**：沉淀的价值来自质量，不来自数量。宁可少写精华，不要多写废话。一次沉淀 200-500 字是正常的，超过 1000 字要想想是否可以更精练。

**不打扰原则**：CHECK 模式的结论如果是"不需要建项目"，静默跳过，不要告诉用户"我判断这不值得记录"。只在需要建项目或需要沉淀时才开口。

**及时性**：在 CAPTURE 发生时立即写文件，不要等到会话结束——那时候细节已经模糊了。

**与 memory 的分工**：
- `memory/long/` → 系统级规律、跨项目通用经验（由 health-check 维护）
- `knowledge/projects/` → 项目级的具体记录、决策上下文（本 skill 维护）

## 示例：何时开口

**应该说**（会话第 10 轮，修复了一个 Binance API 的坑）：
> "这次调试发现了一个有价值的规律，我来把它记录到 `fin-auto-trader` 项目里。"

**不应该说**（用户问了两个关于 Python 语法的问题）：
> 静默。这不值得一个项目文件。

**应该说**（会话一开始，用户说"我想做一个新的 MCP server"）：
> "这是个新项目，我在 `knowledge/projects/mcp_server/` 建立跟踪文件，边做边记录。"
